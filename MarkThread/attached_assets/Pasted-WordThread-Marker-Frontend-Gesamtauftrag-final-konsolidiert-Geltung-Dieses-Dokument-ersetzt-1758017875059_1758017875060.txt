WordThread / Marker-Frontend — Gesamtauftrag (final, konsolidiert)

Geltung: Dieses Dokument ersetzt alle vorherigen Teilpläne. Später Gesagtes hat Vorrang. Ziel ist ein vollständig lauffähiges Frontend inkl. Logik, Anbindung und CI/CD, nach FrontCollabRepoSetup.

1. Ziel und Scope

Eigenständiges Frontend-Modul der Marker-Engine.

Offline-first via Electron. Browser-Fallback mit File-Upload.

Sichtbare, performante UI im WordThread-Design (Glow, Gradient, volumetrisches Licht).

Strikte Einhaltung der Marker-Architektur ATO → SEM → CLU → MEMA und Intuitions-CLUs, keine Schemaänderung notwendig.

Validierung gegen AnalysisBundle v1.0 und PromatraHandshake v0.2.

Keine Mockups in Produktion. PNG- und JSON-Export integriert.

2. Terminologie (verbindlich)
Begriff	Definition
Marker-Logic	Vorverarbeitung von Language-Patterns (YAML/JSON auf MongoDB), gesteuert durch Schema- und Gewichtungsdateien.
Spark-Service	Batch-Verarbeitung der Marker-Logic, semantisches Reasoning.
shakehands	Übergabe-Schnittstelle zwischen Marker-Logic und Spark-Service.
Lean‑DB 3.4	Konventionen für Marker-Schemas, Strukturen, Validierung und CI.
Intuitions‑Marker (CLU_INTUITION_*)	Zustandsbehaftete CLUs, die Familien von SEM bündeln; Laufzeit-States provisional → confirmed → decayed inkl. Multiplier und Telemetrie.

Hinweis: SEM benötigen ≥2 distinkte ATO in composed_of. Pro Marker genau ein Strukturblock: pattern oder composed_of oder detect_class. Beispiele ≥5. CI prüft dies.

3. Architektur

Schichten: Ingest → Validation → Store → Selectors → Worker‑Renderer → UI.
Laufzeiten: Primär Electron (Node‑FS), optional Browser (File‑Upload).
Datenweg: Input: AnalysisBundle(JSON)/SQLite, optional Handshake. Output: Canvas‑Render, PNG‑Snapshot, JSON‑Sichtzustand.

Marker‑Kontext: Vier Ebenen (ATO/SEM/CLU/MEMA). ATOs sind reine Beobachtung, Bedeutung entsteht über Kombinationen. UI muss diesen Bottom‑up‑Pfad sichtbar machen.

Backend‑Referenz: Marker- und Event‑Collections (atomic/semantic/cluster/meta) und Zeitachsen‑Indizes sind gesetzt. Frontend spiegelt diese Perspektiven in Filtern/Selectoren.

4. Datenmodelle & Verträge
4.1 AnalysisBundle v1.0 (JSON)

Felder: bundle, context, input, hits[], aggregates, scores, drift, provenance.
Zod‑Pflicht: Schema strikt; invalid → Abbruch mit Meldung.

4.2 Dash‑Out‑Sys (SQLite)

Tabellen: messages, hits, aggregates, scores, drift_axes, provenance. Pflichtindizes für Zeit/Conversation.
Electron: native SQLite. Browser: WASM/Worker.

4.3 PromatraHandshake v0.2

Nur über Attachments/Evidence referenziert.

4.4 Marker‑Vorlagen (Referenz für UI/Evidence)

ATO: Regex/Token + Beispiele.

SEM: composed_of ≥2 ATO, Aktivierungsregel.

CLU: Aggregation über Fenster, activation, scoring.

MEMA: Regel- oder Detector‑basiert, genau ein Strukturblock.

Schema‑Gewichte & Score‑Fenster sind darstellbar (KPI/Overlay).

Intuitions‑CLUs (familienbasiert, runtime‑zuständig). UI zeigt State, Multiplier, Telemetrie.

5. Funktionale Anforderungen (FRD)

Ingest

FSAdapter: Beobachten von OUTBOX/*.json und *.sqlite.

HttpAdapter: optionale /api/v1/... Endpunkte.

Gemeinsame IIngestAdapter‑Schnittstelle.

Validation

Zod‑Schemas für AnalysisBundle v1.0 und PromatraHandshake v0.2.

Bei Fehler: Abbruch + klare Fehlermeldung.

Store

Append‑only. Kein Mutieren des Original‑Bundles.

Selectors: Top‑Marker, Marker→Hits, Zeitbereich, Drift‑Achsen.

Rendering

OffscreenCanvas im Worker. Fallback: ImageBitmap.

Charts: LineChart, Heatmap, DriftChart.

LTTB‑Downsampling, Virtualisierung ≥10k Rows.

UI

Header/KPIs; Top‑Marker‑Tabelle; Drift‑Panel; Detail‑Hits; Evidence‑Panel; Provenance‑Log.

Export: PNG‑Snapshot des Canvas; JSON‑Export des Sichtzustands.

6. Nicht‑funktionale Anforderungen (NFR)

Performance: First Paint < 500 ms bei Bundle ≤ 10 MB. Interaktion ≥ 30 FPS.

Availability: Offline‑first (Electron). Browser‑Build mit File‑Upload.

Security/Integrity: SHA‑256‑Digests sichtbar. Append‑only. Keine Mutation.

Quality Gates: pre‑commit: lint, typecheck, tests, no‑mockups‑check.

14. Zod‑Skelett (Validation)
// schema/analysisBundle.ts
import { z } from "zod";

export const Hit = z.object({
  id: z.string(),
  marker_id: z.string(),         // ATO/SEM/CLU/MEMA
  ts: z.number(),
  conv: z.string().optional(),
  payload: z.record(z.any()).optional(),
});

export const AnalysisBundle = z.object({
  bundle: z.string(),
  context: z.record(z.any()).optional(),
  input: z.record(z.any()).optional(),
  hits: z.array(Hit),
  aggregates: z.record(z.any()).optional(),
  scores: z.record(z.any()).optional(),
  drift: z.record(z.any()).optional(),
  provenance: z.record(z.any()).optional(),
}).strict();
